关于SpringIoC容器的理解

Created2023-11-20|Updated2023-11-21
|Post View:

IoC容器的大致实现框架：

我们用一个部件来对应 Bean 内存的映像

一个 XML reader 负责从外部 XML 文件获取 Bean 的配置

一个反射部件，负责加载 Bean Class 并且创建这个实例

创建实例之后，我们用一个 Map 来保存 Bean 的实例

提供一个 getBean() 方法供外部使用

IoC容器的核心就是BeanFactory的实现类比如SimpleBeanFactory。所有的Bean实例都是存储在SimpleBeanFactory内的Map里的，需要的时候将其取出来

自定义ClassPathXmlResource负责拿到xml里的内容，交给XmlBeanDefinitionReader解析，解析完并加载到xmlBeanDefinitionReader里的Beanfactory中

以上过程全部发生在ClassPathXmlApplicationContext中，

首先在ClassPathXmlApplicationContext的构造方法里新建一个beanfactory实例，然后将这个实例传入XmlBeanDefinitionReader中进行bean.xml的解析，并且将bean实例注入这个beanfactory，最后将这个beanFactory传给全局变量beanFactory

完成了原始IoC容器后，开始以下任务：

增加单例 Bean 的接口定义，然后把所有的 Bean 默认为单例模式

预留事件监听的接口，方便后续进一步解耦代码逻辑

扩展 BeanDefinition，添加一些属性

首先构建单例Bean

定义了一个默认单例Bean注册类实现单例Bean注册接口

用默认Bean工厂实现这个单例Bean注册接口，

其中，defaultSingletonBeanregistry通过维护一个singletonObjects列表来保证spring创建的bean都是默认单例的

接着添加事件监听？？？不懂这个是有什么用处。

添加了ArgumentValue，PropertyValue

主要处理任务都在XmlBeanDefinitionReader中

处理 主要用loadBeanDefinitions这个函数

loadBeanDefinition遍历传入的xml文件的resource抽象类，（被自定义ClassXmlResource修饰过），从中获取id，class，setter注入属性：property，构造器注入属性：constructor-arg，用来定义BeanDefinition并且将其注册到SimpleBean工厂，SimpleBean工厂继承了默认单例bean注册中心DefaultSingletonbeanRegistry，其中有个ConcurrentMap用来存放beanid和Object，SImpleBeanFactory使用createBean创建Bean。

属性主要有三个参数：Type，例如String，Name，例如Address，Value：例如 huadeli

简要说一下property和args的存储，定义了两个类，分别代表一组属性，再定义一个集合，存储这些属性组，类似数据库中的表。

loadBeandefinition方法将bean的id，class，property，args等注册进去，等以后使用

Bean工厂中会有一个registerBeanFactory的方法，将beandefinition和beanid注册进定义集合，方便后续createbean使用

CreateBean流程：

  1. 找到BeanClass
  2. 将Setter和Constructor参数找出来并注入？

解析arg和property的流程：

arg：由于是构造器参数，所以不是很注重参数的名字，只需要知道类型和值将其注入即可。

property：需要使用obj本身的setter方法，使用反射拿到obj的setter方法，再进行调用。完成属性注入

在beandefinition的注册过程中，会在处理property的时候将ref拆分成，value和isref，如果isref是true，那么value就是beanName

spring解决循环依赖问题：

使用了三级缓存，分别是singletonObjects，保存所有已经创建完成的Bean，singletonFactories，正在注入属性的Bean，earlySingletonObjects，初始化Bean（毛胚Bean）

半成品Bean会进行构造器注入，然后作为一个毛胚，充当其他Bean的属性，引用对象。

增强IoC—支持注解开发

注解解释的时机是在createBean之后。initbean之前

创建一个扩展修改Bean接口：BeanPostProcessor，其中有两个方法，分别是bean初始化之前的处理和Bean初始化之后的处理

beanpostprocessor before initialization

通过反射的方式判断属性是否有autowired注解， 如果有就创建这个注解修饰对象的bean然后再利用反射（field的set方法）将这个对象替换为初始化完成的bean

之后新增abstractBeanfactory，其中提供了refresh getbean registerBeanDefinition的默认实现方法

refresh就是将每个bean实例化获取一遍

getbean就是创建bean实例

getbean中的两个抽象方法都是交由具体子类去实现的

注解注入完整过程“

  1. 启动ClassPathXmlApplication容器，执行refresh方法
  2. refresh中会调用registerBeanPostProcessors，注册bean后处理器，目前有的就是AuworiedAnnotationBeanPostProcessor
  3. 执行onRefresh，执行AbstractBeanFactory的refresh方法
  4. ab中的refresh会获取bean定义，执行getBean创建好bean实例
  5. getBean创建完Bean实例后会调用Bean处理器并初始化
  6. 初始化中有前置，过程，后置处理器，调用完成注解的注入
  7. 实现准备好的autowired后处理器会解释Bean中的Autowired注解

构建工厂体系并添加容器事件

  1. 增强扩展性：
    1. ListableBeanFactory
    2. ConfigurableBeanFactory
    3. ConfigurableListableBeanFactory
    4. EnvironmentCapable
  2. 实现DefaultListableBeanFactory，SpringIoC的引擎
  3. 改造ApplicationContext

扩展性

ListableBeanFactory：

将 Factory 内部管理的 Bean 作为一个集合来对待，获取Bean 的数量，得到所有 Bean 的名字，按照某个类型获取 Bean 列表等等。这个特性就定义在 ListableBeanFactory 中

ConfigurableBeanFactory

维护Bean之间的关系，支持Bean处理器看作独立的特性，定义在ConfigurableBeanFactory接口中

ConfigurableListableBeanFactory

将AutowireCapableBeanFactory、ListableBeanFactory 和 ConfigurableBeanFactory合并在一起

思考

1.控制反转，反转了什么？

传统程序设计中，需要程序员手动的new一个对象，同时为其注入属性，如果其依赖了其他的对象，会增加代码耦合度

IoC容器解决了这个难点，一是可以自动创建对象需要的实例，二是结局了依赖注入，自动将依赖方注入到Bean中

这样可以简化对象创建与依赖关系的管理，使代码更加易于维护和扩展

2.构造器注入和Setter注入的异同？

相同之处：

都是为了将依赖对象传递给类或对象，减少运行时对象之间的依赖关系，都可以减少类与依赖对象的耦合度，让代码更加灵活

不同之处：

  1. 构造器注入

    1. 优点：

      能确保类的依赖完整，因为依赖对象需要在构造器中传入，如果没有传入完整的依赖对象，类的实例化就会失败。 构造器注入代码结构简洁，易于理解和维护。 适用于生命周期较短的对象，对象在实例化时会获得所需的依赖，对象销毁时也会自动释放。

    2. 缺点：

      如果依赖对象较多，构造函数的参数列表可能会很长，这会让代码变得复杂难懂。 不够灵活: 使用构造器注入，类的依赖对象需要在构造器中进行声明和初始化，这会导致类与依赖对象之间的耦合度比较高，而且在运行时无法更改依赖对象。 如果对象的生命周期较长，构造器注入就不太适用了，因为依赖对象可能会在对象使用过程中发生变化，但是构造器注入无法满足这类需求。

  2. Setter注入

    1. 优点：

      比较灵活，Setter注入能够让开发者更灵活地修改类的依赖关系。 Setter注入使得代码易于测试，因为它可以很容易地mock或替换依赖对象，帮助开发者做单元测试 代码简洁: Setter注入避免了构造函数中出现过多参数的情况，使得代码更加易于理解和维护。 容错性强: Setter注入使得依赖对象和被依赖对象之间的耦合度降低，即使某些依赖对象无法使用，也不会导致整个系统崩溃。

    2. 缺点：

      为每个依赖属性定义对应的setter方法，需要使用大量的setter方法，这可能会让代码显得冗余。 对象状态不一致，由于Setter注入是通过调用一系列方法来完成依赖注入的，依赖对象可能被注入了一部分但未完成全部注入的情况，从而导致对象状态不一致。 可能会增加耦合度，通过Setter注入，类的实例可能需要访问依赖类的属性或方法，增加依赖对象之间的耦合度。

结论：反射技术是IoC容器赖以工作的基础

解决Bean之间的依赖

增加ref属性表示引用

继续在xmlBeanDefinitionReader中处理ref属性

程序解析 标签后，获取了 ref 的参数，设置了 isRef 的值，把它添加到了 PropertyValues 内，最后程序调用 setDependsOn 方法，它记录了某一个 Bean 引用的其他 Bean。

改造createBean方法：

改造handleProperties方法：通过属性判断，进行必要的getBean，调用setter注入

paramTypes[0] = Class.forName(pType); paramValues[0] = getBean((String)pValue);

这样一来，如果有多级引用，就会形成一个多级的 getBean() 调用链。

由于在调用 getBean()的时候会判断容器中是否包含了 bean instance，没有的话会立即创建，所以 XML 配置文件中声明 Bean 的先后次序是任意的。

循环依赖问题：

在注入属性之前，其实这个 Bean 的实例已经生成出来了，由于bean是唯一的，即使Java传递的是对象地址，所以即使后续更改bean的属性也不会陷入死循环

因此决定，引入一个结构：earlySingletonObjects

  • 循环依赖：

    第一步，先实例化 ABean，此时它是早期的不完整毛胚实例，好多属性还没被赋值，将实例 放置到 earlySingletonObjects 中备用。然后给 ABean 注入属性，这个时候发现它还要依赖 BBean。

    第二步，实例化 BBean，它也是早期的不完整毛胚实例，我们也将实例放到 earlySingletonObjects 中备用。然后再给 BBean 注入属性，又发现它依赖 CBean。

    第三步，实例化 CBean，此时它仍然是早期的不完整的实例，同样将实例放置到 earlySingletonObjects 中备用，然后再给 CBean 属性赋值，这个时候又发现它反过来还要 依赖 ABean

    第四步，我们从 earlySingletonObjects 结构中找到 ABean 的早期毛胚实例，取出来给 CBean 注入属性，这意味着这时 CBean 所用的 ABean 实例是那个早期的毛胚实例。这样就 先创建好了 CBean。

    第五步，程序控制流回到第二步，完成 BBean 的属性注入

    第六步，程序控制流回到第一步，完成 ABean 的属性注入。至此，所有的 Bean 就都创建完 了。

需要修改getBean，createBean，doCreateBean

getBean中增加 从earlySingletonObjects里获取实例的判断

createBean中增加 创建毛胚bean实例的步骤doCreateBean，存放进入毛胚实例缓存，处理属性

doCreatebean中，仅调用构造方法，不进行属性处理

get ABean(create Early) → need BBean → get BBean(create Early) → need CBean → getCBean (create Early)→ need ABean → get ABean → return CBean → return BBean → create ABean

增加了 lazyInit 属性，表示 Bean 要不要在加载的时候初始化，

dependsOn 属性记录 Bean 之间的依赖关系，

最后还有构造器参数和 property 列表

新增BeanDefinitionRegistry接口，存放BeanDefinition

提供register、remove、get、contains

调整 BeanFactory，新增 Singleton、Prototype 的判断，获取 Bean 的类型

调整SimpleBeanFactory 实现了 BeanDefinitionRegistry，这样SimpleBeanFactory 既是一个工厂同时也是一个仓库

调整ClassPathXmlApplicationContext

包装方法 refresh()，对大多数不需要懒加载的bean在容器启动的时候就进行初始化

在SimpleBeanFactory中创建该方法：遍历BeanName，getBean

在ClassPathXmlApplicationContext的构造方法中调用refresh

开发注解@Autowired

这个注解是作用在一个实例变量上的，为了生效，我们首先必须创建好这个对象，也就是在 createBean 时机之后。

定义两个接口，bean初始化前后

定义autowired注解，修饰成员变量，运行时生效

利用反射获取对象的field进行注入

Autowired的处理类

AutowiredAnnotationbeanPostProcessor implements BeanPostProcessor

重写bean初始化前处理器

查找所有field，找到autowired注解修饰的，拿到fieldname，getbean，注入属性

引入了autowireCapableBeanfactory，专门为autowired注入的bean准备的

抽象出abstractBeanFactory

内置默认的beandefinition names earlySingletonobjects

实现了默认的bean管理方法：getBean，createBean，doCreateBean

预留两个抽象方法beanPostProcess方法给子类实现

实现AutowireCapableBeanFactory

用一个列表 beanPostProcessors 记录所有的Bean 处理器，这样可以按照需求注册若干个不同用途的处理器，然后调用处理器

调整 ClassPathXmlApplicationContext，引入的成员变量由 SimpleBeanFactory改为新建的 AutowireCapableBeanFactory，并在构造函数里增加上下文刷新逻辑

启动 ClassPathXmlApplicationContext 容器，执行 refresh()

refresh 执行过程中，调用 registerBeanPostProcessors()往 BeanFactory 里注册Bean 处理器，如 AutowiredAnnotationBeanPostProcessor

执行 onRefresh()， 执行 AbstractBeanFactory 的 refresh() 方法。

AbstractBeanFactory 的 refresh() 获取所有 Bean 的定义，执行 getBean() 创建 Bean 实例。

getBean() 创建完 Bean 实例后，调用 Bean 处理器并初始化

applyBeanPostProcessorBeforeInitialization 由具体的 BeanFactory，实现

事先准备好的 AutowiredAnnotationBeanPostProcessor 方法里面会解释 Bean 中autowired注解

构建工厂体系 添加容器事件

新增四个接口：

ListableBeanFactory ConfigurableBeanFactory

ConfigurableListableBeanFactory EnvironmentCapable

实现DefaultListableBeanFactory，作为SpringIoC的引擎

ListablebeanFactory：将Bean作为集合来对待，获取bean数量，得到bean名字，按照类型获取bean列表

ConfigurableBeanFactory：维护了bean之间的依赖关系，支持bean的处理器

ConfigurableListableBeanFactory：集大成者，**extends** ListableBeanFactory, AutowireCapableBeanFactory, ConfigurableBeanFactory

interface代表一种特性或者能力，将这些能力一一抽取出来互不干扰

想具备某种能力，需要去实现这些interface，这就是interface segregation（接口隔离原则）

继续将autowireCapableBeanFactory抽象，改造成一个接口，使其具备bean初始化前后处理的能力

新增抽象类 AbstractAutowireCapableBeanFactory 替代原有的实现类

**extends** AbstractBeanFactory **implements** AutowireCapableBeanFactory
